Despiece de SHOW, HP0 y signal aspect

Esta sección baja del callback de aplicación al struct, al enum, al payload SCI-LS y al telegrama SCI que finalmente viaja como datos de aplicación RaSTA.

1. Creación del aspecto por defecto

scils_signal_aspect_defaults() reserva memoria con rmalloc(sizeof(scils_signal_aspect)) y rellena los diez campos del struct. El comportamiento probado en el repositorio deja conceptos apagados o sin información, y dark_switching en SCILS_DARK_SWITCHING_SHOW.

Evidencia: src/sci/c/scils_telegram_factory.c:6 y src/sciTest/c/scilsTests.c:8.

2. Conversión de default a HP0

El callback onHandshakeComplete() no construye un struct desde cero. Primero toma los defaults y luego solo cambia signal_aspect->main a SCILS_MAIN_HP_0. El resto queda igual: additional, ZS3, ZS3v, ZS2, ZS2v apagados; driveway sin información; dark switching en SHOW.

Evidencia: src/scils_example/c/main.c:34 a src/scils_example/c/main.c:37.

3. Valor real de SCILS_MAIN_HP_0

SCILS_MAIN_HP_0 es un valor del enum scils_main y vale 0x01. En este PoC no se demuestra aquí una semántica normativa externa; el código lo trata como byte de concepto principal que se copia a payload.data[0] y después se imprime como MAIN = 0x01.

Evidencia: src/sci/headers/scils_telegram_factory.h:36.

4. API de envío scils_send_show_signal_aspect

scils_send_show_signal_aspect(ls, "S", signal_aspect) es una fachada fina: usa ls->sciName como sender local, conserva el receiver recibido y delega la construcción a scils_create_show_signal_aspect(). Después entrega el telegrama a scils_send_telegram().

Evidencia: src/sci/c/scils.c:101.

Cabecera SCI común

Antes del payload, sci_create_base_telegram() fija SCI_PROTOCOL_LS = 0x30, el tipo de mensaje, sender y receiver. Los nombres SCI tienen 20 bytes y se rellenan con 0x5F si son más cortos.

Bytes enviados a RaSTA

sci_encode_telegram() concatena 43 bytes de cabecera SCI y el payload usado. Para SHOW con HP0/defaults, la longitud SCI observable es 52 bytes: 43 de cabecera y 9 de payload.

Recepción y parseo

El servidor recibe bytes RaSTA, sci_decode_telegram() reconstruye el telegrama, scils_on_rasta_receive() detecta SCILS_MESSAGE_TYPE_SHOW_SIGNAL_ASPECT y handle_show_signal_aspect() parsea los 9 bytes antes de invocar onShowSignalAspect().

STATUS de retorno

La demo del servidor no valida ni actúa sobre campo físico. Devuelve el mismo signal_aspect recibido como SIGNAL_ASPECT_STATUS, pero la factory STATUS solo serializa 7 bytes, por lo que no transporta deprecation_information ni driveway.

Red remota y sustitución de IPs

El repositorio local conserva 10.0.0.x en los .cfg. Docker los transforma a 10.10.10.x durante el build.

Servidor remoto

  • Contenedor: rasta-server.
  • IP Docker: 10.10.10.100/24.
  • Config final: rasta_server.cfg.
  • Canales: 10.10.10.100:8888 y 10.10.10.100:8889.

Cliente SCI-LS

  • Contenedor: rasta-client1.
  • IP Docker: 10.10.10.110/24.
  • Config final: rasta_client1.cfg.
  • Destino: SERVER_CH1 y SERVER_CH2 leídos con getenv().

Observación de redundancia

  • El script asigna la misma IP a canal 1 y canal 2 de cada contenedor.
  • La separación observable en este Docker es por puerto, no por interfaces/redes independientes.
  • Para ensayos de degradación realista habría que separar redes/canales.
