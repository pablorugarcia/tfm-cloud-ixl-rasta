Modo presentacion

Del arranque al cleanup, como una historia interactiva.

Los pasos principales se muestran como escenas: establecimiento PDI, callbacks, bucle de ruta, envio SCI-LS y confirmacion final. La documentacion detallada sigue debajo, con busqueda y filtros.

Alcance: es una documentacion del PoC de laboratorio segun el codigo inspeccionado. No es una implementacion normativa ni evidencia de cumplimiento SIL4, EN 50159, EULYNX o RaSTA. Las capas SCI-LS, RaSTA y el ICD se muestran como dependencias de ejecucion del Cloud IXL.

Resumen ejecutivo del flujo

main() mezcla ejecucion secuencial con eventos asincronos. La parte secuencial configura RaSTA/SCI-LS, inicia la conexion, espera el establecimiento PDI y luego procesa ordenes de ruta. La parte asincrona completa el PDI y confirma comandos mediante callbacks.

3fases principales
4callbacks RaSTA registrados
4callbacks SCI-LS registrados
6salidas terminales relevantes

1. Arranque y PDI

  1. Declara estado Cloud IXL, receptor, handle RaSTA y dos canales localhost.
  2. Inicializa RaSTA con sr_init_handle() y registra callbacks.
  3. Inicializa SCI-LS con nombre IXL_CENTRAL y mapea LS_OC a 0x61.
  4. Lanza sr_connect() y espera a PDI_ESTABLISHED o PDI_FAILED.

2. Establecimiento asincrono

  1. on_rasta_handshake() valida el RaSTA ID remoto.
  2. Envia version request SCI-LS.
  3. on_rasta_receive() parsea version response, status begin, status finish y errores.
  4. La condicion se desbloquea cuando el estado llega a PDI_ESTABLISHED o PDI_FAILED.

3. Bucle de ordenes

  1. Lee request, release o quit.
  2. Calcula decision de ruta, traduce a aspecto de senal y construye vector esperado.
  3. Envia telegrama SCI-LS sobre RaSTA y espera confirmacion.
  4. Si la confirmacion coincide, continua; si falla, sale con error y limpia recursos.

Flujo paralelo: mensajes de version, PDI y status

Esta vista sigue el establecimiento PDI completo observado en el repositorio: conexion RaSTA, intercambio de version SCI-LS, solicitud de estado inicial y transicion final a PDI_ESTABLISHED. No introduce nuevos tipos de telegrama.

0x2400 VERSION_REQUEST

Cloud IXL pregunta al OC si acepta la version PDI que quiere usar.

0x2500 VERSION_RESPONSE

El OC contesta con version solicitada, version soportada y resultado.

0x2100 STATUS_REQUEST

Cloud IXL pide la foto inicial del estado del objeto.

0x2200 STATUS_BEGIN

Abre el bloque de estado. No es el estado en si: es el inicio de la transferencia.

0x2300 STATUS_FINISH

Cierra el bloque de estado. Significa: "ya no envio mas estados iniciales".

0x2200 abre la ventana. Dentro podrian viajar mensajes concretos como aspecto o luminosidad. 0x2300 la cierra. Por eso no existe un unico STATUS_RESPONSE.

PoC de laboratorio
Lectura correcta del status No hay SCI_MESSAGE_TYPE_STATUS_RESPONSE. El estado inicial se delimita con STATUS_BEGIN -> posibles estados SCI-LS -> STATUS_FINISH.
Ramas de fallo PDI Remote ID inesperado, version incompatible, payload vacio con longitud distinta de cero, estado PDI incorrecto, error de parseo o timeout terminan en PDI_FAILED.
Separacion de capas SCI-LS es el payload/interfaz de aplicacion; RaSTA transporta los bytes; el PDI aqui es la maquina de estados local del PoC que decide cuando Cloud IXL puede enviar ordenes.

Riesgos y limites del flujo actual

Observaciones de diseno derivadas del codigo, sin proponer cambios automaticos.

El PDI depende de callbacks asincronos. main() avanza por condicion, pero los estados se modifican desde callbacks que corren en hilos creados por RaSTA. Por eso el mutex y la condicion son parte central del flujo, no un detalle accesorio.
La confirmacion de aspecto usa un camino especial. Durante PDI_WAIT_CONFIRMATION, on_rasta_receive() intercepta SCILS_MESSAGE_TYPE_SIGNAL_ASPECT_STATUS, parsea el vector ICD de 18 bytes y compara con expected_signal_vector. Fuera de esa espera, el mensaje cae al manejador generico SCI-LS.
La liberacion solo cambia estado local si hubo confirmacion OK. Si la orden era release y la confirmacion coincide, se llama a release_route(); su retorno se descarta. Eso es un hecho del flujo actual, no una afirmacion de validez operacional.
