Guía interactiva de conceptos esenciales

Esta pestaña interna resume los conceptos de programación, arquitectura y telecomunicaciones que hacen falta para leer el repositorio con criterio. Cada bloque está trazado a archivos concretos y enlaza con el atlas cuando existe un nodo relacionado.

Cómo usar esta guía.

Primero separa capas: SCI-LS define el contenido de aplicación, RaSTA/RaSTA-like transporta bytes con estado de conexión, redundancia/UDP representa canales de red y los ejemplos conectan todo en localhost. Después abre los conceptos o el explorador por archivo para saltar al mapa maestro.

Límite de interpretación.

La guía describe un prototipo de laboratorio. No prueba conformidad SIL4, EN 50159, EULYNX ni RaSTA. Cuando se menciona Cloud IXL u Object Controller es para orientar responsabilidades de TFM, no para afirmar una integración normativa.

Aplicación ejemplo Servidor/cliente localhost

src/examples_localhost/c/scils.c decide rol, registra callbacks y dispara el flujo SHOW/STATUS. Es la capa de demostración, no la especificación del sistema.

SCI-LS Payload e interfaz de aplicación

scils.c convierte órdenes de señal luminosa en telegramas SCI y despacha callbacks como on_show_signal_aspect_received.

SCI común Formato de telegrama

sci.c empaqueta protocol type, message type, sender, receiver y payload en arrays de bytes. Aquí aparecen padding, longitudes y orden de bytes.

RaSTA SR Conexión, secuencia y supervisión

rasta_new.c gestiona estados, ConReq/ConResp, DATA, heartbeat, retransmisión, colas y callbacks. Transporta bytes de aplicación.

Redundancia/UDP Canales de transporte

rasta_red_multiplexer.c, rastaredundancy_new.c y udp.c conectan PDUs RaSTA con sockets, canales redundantes, CRC y diagnóstico.

Configuración y pruebas Reproducibilidad del laboratorio

config/*.cfg, CMakeLists.txt, build.gradle y los tests documentan cómo se compila, con qué parámetros se ejecuta y qué módulos tienen cobertura.

Arranque del rol.

El ejemplo elige servidor o cliente, crea struct rasta_handle h y carga config local. Conceptos: argumentos, constantes, vida útil de variables y configuración.

Inicialización RaSTA.

sr_init_handle() termina en rasta_handle_init(), donde se preparan callbacks, hashing, mux, conexiones y sub-handles.

Conexión y handshake.

El cliente llama a sr_connect(); RaSTA envía ConReq, procesa ConResp y puede disparar onHandshakeComplete desde un hilo.

Orden SHOW SCI-LS.

SCI-LS crea un telegrama SHOW, SCI común lo codifica y RaSTA lo transporta como mensaje de aplicación.

Recepción y STATUS.

RaSTA deja bytes en cola de aplicación; el callback del ejemplo llama a scils_on_rasta_receive(), que decodifica y despacha el callback SCI-LS.

Riesgo temporal.

En el cliente local, sr_connect() aparece antes de scils_init() y del registro "S" -> 0x61. La guía lo marca como limitación del laboratorio.

1. ¿Qué capa posee esta responsabilidad?

Si el código crea o parsea telegramas de señal, estás en SCI/SCI-LS. Si gestiona conexión, secuencias, heartbeat o retransmisión, estás en RaSTA SR. Si abre sockets o canales, estás en redundancia/UDP.

Comprobar en el atlas

Usa los filtros SCI-LS, RaSTA, RedMux/UDP y Config del mapa maestro.

2. ¿La llamada es síncrona o asíncrona?

Los callbacks RaSTA se disparan desde hilos creados con pthread_create. Por eso una función que parece lineal en main() puede competir con un evento de red.

Archivo clave

src/rasta/c/rastahandle.c y la sección Riesgo carrera.

3. ¿Qué dato entra y qué dato sale?

Para cada función, identifica si recibe una estructura, un puntero, un array de bytes o un ID. Después localiza quién libera memoria y quién consume el resultado.

Ejemplo

scils_send_telegram() codifica un sci_telegram, busca un RaSTA ID y llama a sr_send().

4. ¿Qué queda fuera de evidencia?

El repositorio permite estudiar un PoC, pero no demuestra cumplimiento de normas ferroviarias. Para TFM hay que separar hecho documentado, inferencia, propuesta y limitación.

Aplicación práctica

En documentación formal, evita frases como "cumple SIL4"; usa "prototipo de laboratorio" o "implementación PoC no normativa".

Checklist de revisión contra código real

1. Orden de inicialización cliente.
Verifica en src/examples_localhost/c/scils.c si sr_connect() sigue antes de scils_init() y scils_register_sci_name(). Si sí, documenta el riesgo de carrera.
2. Callbacks RaSTA.
Comprueba que fire_on_receive() y fire_on_handshake_complete() crean callbacks en hilos. Ese detalle condiciona el diseño de estado compartido.
3. Mapa SCI -> RaSTA.
Confirma dónde vive scils->sciNamesToRastaIds, cuándo se rellena y qué ocurre si se envía antes de registrar "S" -> 0x61.
4. Colas RaSTA.
Localiza fifo_send, fifo_app_msg y fifo_retr en struct rasta_connection; diferenciar estas colas te evita mezclar payload SCI-LS con PDU RaSTA.
5. Separación de capas.
SCI-LS crea y parsea telegramas de aplicación; RaSTA transporta y protege esos bytes en el ejemplo. No documentes SCI-LS como transporte.
6. Frontera ICD.
Para tu TFM, el ICD debe describir mensajes, IDs, estados, timeouts y trazabilidad commanded/reported entre Cloud IXL y OC, no solo llamadas internas de C.
